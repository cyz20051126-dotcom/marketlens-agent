# MarketLens Agent v2.1 修复完成总结

## 一句话

把 codex 留下的"声称多智能体但实际是死代码"的 agent 项目，改造成真的能跑的多智能体研究系统：Triage/Planner/Extractor/Writer 全调 DeepSeek，Search/Extractor/Verifier/Store 接进主路径，DuckDuckGo 真搜索，98 个测试全绿。

## 修复前 vs 修复后

| 维度 | 修复前 | 修复后 |
|------|--------|--------|
| LLM 接入 | llm.py 写了但 orchestrator/agents 没 import | Triage/Planner/Extractor/Writer 全调 DeepSeek，解析失败降级规则 |
| 多智能体链 | Search/Extractor/Verifier/Store 是死代码 | orchestrator 在证据<2时走完整链：Planner→Search→Extractor→Verifier→Store |
| 网页搜索 | FirecrawlSearchTool 只写 `{status: "prepared"}` JSON | WebSearchTool 调 DuckDuckGo HTML，解析真实结果，失败降级 |
| Writer 输出 | 字符串模板拼接 | LLM 生成中文回答，后处理确保 evidence ID + 局限性 + 非投资建议 |
| 触发搜索 | 关键词触发（"最新""搜索"等） | 本地严格匹配 < 2 条才搜（spec §6 语义） |
| Trace latency | 硬编码 1ms | time.perf_counter 真实计时 |
| Todo board | 全部 "Planned for research escalation" | _complete_todo 反映真实状态 |
| 种子数据 | 会被 demo run 污染 | EvidenceStoreTool 写 work/extracted_evidence.csv，种子不动 |

## 完成的 5 个 Task

1. **增强 LLM client 层** — LLMClient Protocol + DeepSeek 加 temperature/max_tokens + FallbackLLMClient 升级（有 context 时基于 evidence 生成结构化中文）+ MockLLMClient
2. **DuckDuckGo 真搜索** — FirecrawlSearchTool → WebSearchTool，调 DuckDuckGo HTML，失败降级 degraded_fallback
3. **四个 agent 纯 LLM 驱动** — Triage（返回 intent + rewritten_query，避免纯路由 persona）/ Planner / Extractor（research isolation）/ Writer，解析失败降级规则
4. **orchestrator 接齐多智能体链** — Triage→EvidenceSearch→[证据<2: Planner→Search→Extractor→Verifier→Store]→[finance: FinanceLens]→Writer，真实 latency，todo 反映状态
5. **端到端测试** — MockLLMClient + stub search 跑完整链路，验证 agents_invoked/中文回答/evidence ID/trace 顺序/真实 latency

## 关键设计决策（对照 agent-skills 编排模式）

- **反模式 A（路由 Persona）**：Triage 同时返回 intent + query 改写，有领域输出，不是纯路由
- **反模式 B（Persona 调 Persona）**：agent 之间不互调，全是 Python orchestrator 调度
- **反模式 C（改写型顺序编排器）**：orchestrator 是 Python 代码，传原始 dataclass，不做 LLM 改写
- **反模式 D（深层 Persona 树）**：编排深度 = 1
- **模式 5（research isolation）**：Extractor 只返回结构化 JSON，原始 HTML 不进 Writer 上下文
